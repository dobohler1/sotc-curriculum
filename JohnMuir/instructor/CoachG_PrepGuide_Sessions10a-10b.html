Coach G — Sessions 10a–10b Prep Guide

SumoBot Challenge: Build & Program (Session 10a) + Tournament Day (Session 10b) — Everything you need to run both sessions.

1. Overview — What This Unit Is

SumoBot is a two-session capstone unit. Students modify their robots for sumo wrestling, program autonomous search-and-attack behavior using distance sensors, and compete in a double-elimination tournament. This unit applies everything from prior sessions: motor control, sensor reading, while loops, if/elif/else, and custom functions.

SessionThemeWhat Students DoDuration
Session 10a: Build & Program Design + code Physics demo, robot modifications, write/tune sumo code, test matches ~90 min
Session 10b: Tournament Day Competition Final tuning, double-elimination bracket, analysis, awards ~90 min
Why this unit works: Competition is the engagement engine. Every student — even Red Table — has a fighting robot by the end of Session 10a. The differentiation is in code sophistication (Red: tune 3 parameters, Yellow: write sensor logic, Green: build from scratch with edge detection). But everyone competes in the same bracket. That's what makes it motivating.

3. Physical Setup — Do This Before Session 10a

The ring must be ready before Session 10a starts. Students need it for test matches at minute 65. If you're building it during class, you lose testing time and momentum.

Ring Specifications

  • Size: 3 ft diameter (1.5 ft radius) circle
  • Material: White tape on dark floor, or dark poster board with white tape border
  • Starting lines: Two colored tape marks inside the ring, ~10 inches apart, centered
  • Location: On the floor (not a table), with clear space around it for spectators

Test the Red starter code on a real robot before class. Verify it searches and charges. If it doesn't work on your hardware, students will be dead in the water. Common issue: port numbers don't match — adjust the code to match your robots' actual port configuration.

5. Session 10a: Build & Program — Coach G's Playbook

The single most important thing this session: Every team must have a working sumo robot by the test match phase (minute 65). Red Table has complete code — they just need to get it running and tune 3 numbers. If they're still debugging at minute 60, pull them aside and fix it yourself. A non-functional robot means no test matches, no competition buy-in, and a disengaged student in Session 10b.

Before Students Arrive

  • Open Session10a_SumoBot.html in Chrome/Safari, go full screen (F11 or Cmd+Shift+F)
  • Ring taped and ready on the floor
  • Extra LEGO parts on a shared table
  • Code files posted in Google Classroom
  • Worksheets accessible (printed or digital)

Minute-by-Minute

TimeSlideWhat HappensYour Role
0–10 1–5 Physics demo: center of mass (push contest), gear ratios (feel the difference), 3 sumo strategies. Do the push contest with two volunteers. Show gear configurations on a demo robot. Ask each table: "Which strategy will you try?" This is not skippable — the physics concepts are a primary learning objective.
10–15 6–7 Show the ring. Explain rules. 3-second countdown, 60-second limit, fully autonomous, best of 3. Be clear on "fully autonomous." "Once you press play, you cannot touch the robot. If it drives out of the ring, that's on your code."
15–20 8 Spec writing. Students write acceptance criteria on worksheet. Walk the room. Help students be specific: "My robot will spin to search, then charge at 80% speed when it detects something within 30cm."
20–40 9 Build time (20 min). Students modify their driving base for sumo. Circulate. Suggest modifications: front pusher, lower center of mass, wider wheelbase. Checkpoint at minute 30: "10 minutes left on build. Pusher first, aesthetics later."
40–65 10–12 Code time (25 min). Students open their table's file and write/tune code. Direct by table. Red: walk through changing 3 parameters one at a time, test after each. Yellow: check TODOs are filled in correctly. Green: leave them alone unless stuck. See troubleshooting below.
65–80 13 Test matches (15 min). Informal bouts in the ring. Pair up teams. Run 4–5 practice matches. After each: "What would you change?" Give 2 min to tweak between matches. Rotate opponents.
80–90 14–15 Submit code. Name their sumobot. Preview tournament. "Save your code. DO NOT change anything before next session. Name your bot — it needs a name for the bracket."
Have students fill in the "My Parameters" table on the worksheet as they choose values. This slows them down and forces them to think about why they picked a number instead of just mashing random values. It also gives you diagnostic info — if a student's robot isn't working, check their parameter table first.

Session 10a Troubleshooting

ProblemLikely CauseFix
Robot won't detect opponent Wrong port or sensor blocked Check sensor is on Port D (port number 3 in code). Make sure the front pusher isn't blocking the sensor's line of sight.
Robot drives out of ring immediately Motors wired backward or speed too high Swap A and B in motor_pair.pair(). Lower ATTACK_SPEED to 50–60, tune up from there.
Robot just spins and never charges DETECT_DISTANCE too low Increase to 250–300mm. Also check sensor is physically facing forward.
Code crashes on start Python syntax error Check indentation (everything inside while/if must be indented), missing await, wrong port numbers.
Student finished Red code in 5 minutes Not a problem — but redirect them Add a beep when opponent is detected. Try different search/attack speed combos. Add a victory spin. If they're bored, challenge them with Yellow Table's TODO approach.
Green Table stuck on edge detection Color sensor returns unfamiliar values Have them print color_sensor.color(4) first to see the actual values. White is usually 9 or 10. Sensor must face down, within ~1 inch of the floor.
Students spend too long on robot aesthetics Having fun, but not building for function Redirect: "Does it push hard? That's what wins. You can make it look cool after it works."

6. Session 10b: Tournament Day — Coach G's Playbook

The single most important thing this session: Energy. This session lives or dies on your energy as the referee and commentator. Announce play-by-play. Build drama before each match. Make upsets a moment. The students will match your energy — if you're flat, they'll be on their phones. If you're hyped, they'll be on their feet.

Before Students Arrive

  • Open Session10b_SumoBotTournament.html in Chrome/Safari, go full screen
  • Ring is ready (should still be taped from Session 10a, or re-tape)
  • Draw the bracket on the whiteboard or have it printed
  • Timer ready (60-second match timer)
  • Robots and laptops on tables

Minute-by-Minute

TimeSlideWhat HappensYour Role
0–10 1–2 Final tuning. Last chance for parameter tweaks and physical adjustments. "10 minutes. Parameter tweaks only — no new code, no rebuilds." Quick check on Red Table: does their code still run?
10–15 3–4 Quick rules reminder. Bracket reveal. Announce first-round matchups. Explain double elimination: "Lose once, you go to losers bracket. Lose twice, you're out. Everyone gets at least 2 fights." Build anticipation.
15–55 5–10 Tournament matches (40 min). One match at a time, whole class watches. See "Running the Tournament" section below. You are referee AND commentator. Keep it to ~3–4 min per match.
55–65 11–12 Semifinals and Championship. Build maximum hype. Introduce the finalists. Let them talk about their robot. Best of 3 (or best of 5 if time allows). Take photos.
65–75 13 Analysis discussion. "Why did the winning robot win?" Lead whole-class debrief: center of mass, torque, sensor strategy, code strategy. At least 3 students should be able to articulate a physics connection.
75–85 14–15 Awards and reflection. Students complete worksheet. Announce: Champion, Best Design (class vote), Best Strategy (your pick), Most Improved (your pick). Students fill in reflection and self-rating.
85–90 16 Clean up. Robots stored or disassembled. Ring removed. "You designed, built, coded, tested, and competed. That's real engineering."

Running the Tournament

One match at a time, whole class watches. This keeps energy up and prevents chaos. Have the NEXT two teams ready at ringside while the current match runs. You are the referee — your call is final.

Match procedure:

  1. Announce the match: team names and robot names
  2. Both drivers place robots on starting lines
  3. Dramatic countdown: "3... 2... 1... SUMO!"
  4. Both press play simultaneously. Hands off.
  5. Call the play-by-play: "It's circling! Here comes the charge!"
  6. Robot fully outside the ring = that round's loss
  7. 60 seconds with no pushout = closest to center wins the round
  8. Robot stops moving (crash/disconnect) = forfeits after 10 seconds
  9. Best of 3 rounds per match
  10. Update bracket on board. Call next match.

Double-Elimination Bracket

  • Winners bracket: Win and advance. Lose and drop to losers bracket.
  • Losers bracket: Lose here and you're eliminated.
  • Finals: Winners bracket champ vs. losers bracket champ. Losers bracket team must win 2 sets to take the title (they already have one loss).
  • For 6 teams: ~10 total matches. For 8 teams: ~14 total matches.
  • If running long, switch from best-of-3 to single rounds.
Between matches: Ask the winner: "What's your secret?" Ask the loser: "Coming back stronger?" Let losers bracket players make one parameter change before their next match. This keeps eliminated-once students engaged.

Keeping Energy High

  • Announce play-by-play like a sports commentator
  • Let the crowd pick favorites and cheer
  • If a big upset happens, make it a moment: "THE UNDERDOG TAKES IT!"
  • Between matches, let the crowd trash-talk (friendly)
  • Take photos of the finals — parents love this

Quick reference for problems that come up during Session 10b when you can't spend 10 minutes debugging:

8. Quick Reference: Sumo Code

The core sumo loop that all three table levels build toward:

# The sumo brain: search, detect, charge
while True:
    dist = distance_sensor.distance(3)        # Port D

    if dist != -1 and dist < DETECT_DISTANCE:
        # OPPONENT DETECTED -- charge!
        motor_pair.move(motor_pair.PAIR_1, 0, velocity=ATTACK_SPEED)
    else:
        # No opponent -- spin to search
        motor_pair.move(motor_pair.PAIR_1, 100, velocity=SEARCH_SPEED)

    await runloop.sleep_ms(50)

Key Parameters

ParameterWhat It DoesTypical Range
ATTACK_SPEEDCharge speed when opponent detected60–100 (higher = more aggressive but less control)
SEARCH_SPEEDSpin speed while scanning20–40 (lower = more careful scan)
DETECT_DISTANCEHow far away to detect opponent (mm)200–400mm (about 8–16 inches)
