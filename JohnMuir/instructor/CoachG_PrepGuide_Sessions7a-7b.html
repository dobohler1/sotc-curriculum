Coach G — Sessions 7a–7b Prep Guide

Square Up: Build Day (Session 7a) + Race Day (Session 7b) — Everything you need to run both sessions.

1. Overview — What This Unit Is

Square Up is a two-session unit that bridges basic driving code (Sessions 1–2) and sensors (Sessions 8–9). Students program their robot to drive a perfect square with turn signals and beeps, then compete for the fastest clean lap on a taped course.

SessionThemeWhat Students DoAssignment
Session 7a: Build Day Precision driving Write square-driving code, add right_blinker and beep, tune timing Assignment 7: Square Up
Session 7b: Race Day Competition 10-min tuning, then timed runs on the taped course
Why this unit exists: After Sessions 1–2, most students can drive forward and beep. But the jump to sensors is steep — while True: loops and if statements are a lot at once. Square Up gives students more time with motor_pair.move_for_time(), introduces the light_matrix in a fun way (turn signals), and builds the iterate-test-iterate habit they'll need for sensors. The competition makes it worth caring about precision.

3. Course Setup — Do This Before Session 7b

Set up the course before students arrive for Session 7b. If you tape it during class, you lose 5–10 minutes and students start wandering. For Session 7a, students just need open floor space. The course only matters for Session 7b.

Course Specifications

  • Size: 4 ft × 4 ft square
  • Lane width: 1 foot (tape both the inner and outer boundary)
  • Material: Painter's tape or masking tape on the floor
  • Start/Finish: Mark one corner with a different color tape or a piece of paper labeled START

Taping Guide

  1. Tape the outer boundary: a 4 ft × 4 ft square on the floor
  2. Tape the inner boundary: a 2 ft × 2 ft square centered inside, creating a 1-foot lane on all sides
  3. Mark the START corner (top-left is easiest — robot faces right along the first straight)
  4. Make sure there's clear space around the outside of the course — robots that leave the lane will keep driving
Pro tip: Tape the course in an area where students can gather around and watch. The competition is the engagement driver for Session 7b. If the course is hidden in a corner, you lose the audience energy.

4. Session 7a: Build Day — Coach G's Playbook

The single most important thing this session: Students need to leave with working code that drives a square (or at least 2 sides). If they don't have code saved by the end of Session 7a, they'll spend all of Session 7b building instead of competing. Push them to save their code in Google Classroom even if it's not finished.

Before Students Arrive

  • Open Session7a_SquareUp.html in Chrome/Safari, go full screen (F11 or Cmd+Shift+F)
  • Have robots charged and ready on tables
  • Clear floor space for testing — students will be running robots on the floor a lot this session
  • Post Assignment 7 in Google Classroom if not already done

Circulation Priority During Work Time

  1. Red Table first 10 minutes: These students need help getting the starter code typed and running. Common issues: missing import runloop, wrong motor ports, indentation errors. Stand behind them and catch typos as they type.
  2. Yellow Table quick checks: They should be self-sufficient on the basic code. Check in to make sure they're copy-pasting the drive-turn pattern for all 4 sides and adding the blinker function. The main question they'll have: "Where do I put the right_blinker function?" Answer: above async def main():.
  3. Green Table — set and forget: Tell them to use a for loop. If they don't know how: for i in range(4): with the one-side pattern indented inside. Then leave them alone.
  4. Back to Red Table: By minute 25, Red Table should have at least one side driving. If not, diagnose: is it a code error (won't run) or a tuning error (runs but wrong)?

Where to Put the Blinker Function

Students will ask this. The answer:

from hub import light_matrix, sound
import motor_pair
import runloop

motor_pair.pair(motor_pair.PAIR_1, 0, 1)

# Define the blinker function HERE (before main)
async def right_blinker():
    for y in range(5):
        light_matrix.set_pixel(4, y, 100)
    await runloop.sleep_ms(500)
    light_matrix.clear()

# Then main uses it
async def main():
    # ... drive, await right_blinker(), beep, turn ...

runloop.run(main())
If a student puts right_blinker() inside main(): It won't cause an error (Python allows nested functions), but it's messy and confusing. Move it above main() for clarity. If they call it without await, the blinker won't actually wait — the robot will signal and immediately start turning. Make sure they write await right_blinker().

Session 7a Troubleshooting

ProblemLikely CauseFix
Robot drives but doesn't turn Student only has one copy of the drive code, no turn code yet They need to add the turn line after the drive line. Show them slide 4 again.
Robot turns but not 90° Turn time needs tuning Start at 800 ms with steering 100 and velocity 30. Adjust time up or down in increments of 100ms. Every robot is different.
Robot doesn't drive straight Motor speed difference between wheels Change steering from 0 to 2 or -2. Small adjustments. Or check that wheels aren't rubbing on the chassis.
Square is more like a spiral Turns are slightly less than 90° Increase turn time. This is the most common issue. Even 50ms makes a visible difference over 4 turns.
"right_blinker is not defined" Function defined after main(), or typo in name Move the function definition above async def main():. Check spelling.
Blinker shows but robot turns immediately Missing await before right_blinker() Must be await right_blinker(), not just right_blinker().
"I pasted it 4 times but it only drives one side" Pasted outside the main() function (no indentation) All 4 copies of the drive-turn block need to be indented inside async def main(): (4 spaces).
Code runs but nothing happens Missing runloop.run(main()) at the bottom Add it. Must be at the bottom, no indentation.

5. Session 7b: Race Day — Coach G's Playbook

The single most important thing this session: Every student makes at least one official run on the course. Even if their code isn't perfect, they bring their robot to the line and press play. A DNF (did not finish) is infinitely better than "I didn't try." The energy of this session comes from the whole class watching each run. Keep the audience engaged.

Before Students Arrive

  • Open Session7b_SquareUpRace.html in Chrome/Safari, go full screen
  • Course must be taped and ready (see Section 3 above)
  • Have a stopwatch ready (phone timer works)
  • Have robots charged and on tables
  • Print the scoring sheet below or have it on your phone
  • Clear chairs/desks away from the course so students can gather around

Minute-by-Minute

TimeSlideWhat HappensYour Role
0–3 1–2 Title + show the course. Walk everyone over to see it. "This is the course. 4 feet by 4 feet. 1-foot lane. Stay in the lane or you're eliminated." Let them see it and get excited.
3–5 3 Rules: clean run (stays in lane, signals, beeps, completes square) vs. eliminated (leaves lane, missing signal, stops mid-course). Be clear. "Any wheel outside the tape = that run doesn't count. Missing a beep or blinker = doesn't count. But you get unlimited attempts."
5–15 4 Tuning time. Students open their code from Session 7a and test on the open floor. Circulate. Help anyone who lost their code (have them check Google Classroom for their saved draft). Key question for everyone: "Is your square still a square?" Then: "Is it tight enough for the 1-foot lane?"
15–17 5 Explain the race procedure. Call names, place at start, press play, hands off. "When I call your name, bring your robot to START. Place it facing the first straight. Press play and step back. No touching once it starts."
17–40 6 Official runs. Call students one at a time. Time each run. Record results on the leaderboard. See "Running the Race" section below.
40–45 7–8 Submit final code. Announce winner. Closing. Everyone submits Assignment 7 with their final code. Celebrate the winner. Close with: "You tuned a robot to navigate a course. That's engineering."

Running the Race

Pace is everything. You have about 23 minutes for runs. With 10–14 students and unlimited attempts, you need to keep things moving. Call the next student while the current one is retrieving their robot. Don't let anyone hog the course.

Race procedure for each run:

  1. Call the student's name
  2. Student places robot at START, facing the first straight
  3. Student presses play and steps back — hands off
  4. Start your stopwatch when the robot begins moving
  5. Watch for: wheels inside tape, blinker before each turn, beep before each turn
  6. If the robot leaves the lane: Call "Out!" — that attempt is done. Student retrieves robot and can get back in line.
  7. If the robot completes the square: Stop the timer. Record the time. Announce it to the class.
  8. Student retrieves robot. Call the next name.

Managing the queue:

  • Go through the roster once (everyone gets their first attempt)
  • Then open it up: anyone who wants another attempt raises their hand
  • Students waiting should be at their laptops tuning their code, not crowding the course
  • If someone keeps failing, encourage them: "Go adjust your turn time by 50 milliseconds and try again"
Announce times to the class. "Vincent: 14.3 seconds! That's the time to beat." This creates natural competition and keeps the audience watching. Update the leaderboard (slide 6) live if you have it projected, or just announce verbally.

What Counts as a Clean Run

RequirementWhat to Watch For
Stays in lane All wheels inside the tape for all 4 sides. Be strict — if you let one slide, everyone will argue.
Blinker before each turn Right column of the 5×5 matrix lights up before each of the 4 turns. Doesn't need to be long — just visible.
Beep before each turn Audible beep before each turn. If the room is loud, stand close and listen.
Completes full square Robot drives all 4 sides and returns to approximately where it started.

6. Managing Engagement Across Both Sessions

Session 7a: The "I'm Done" Problem

Some students will get the basic square working in 15 minutes and say "I'm done." They're not. Push them to:

  1. Add the blinker and beep (required for a clean race run)
  2. Tune for speed — higher velocity = shorter race time, but harder to stay in the lane
  3. Test on a measured 4-ft stretch — put tape down for one side and see if their robot stays in a 1-foot lane
  4. Use a for loop (Green Table) — cleaner code and easier to tune since they only change the numbers once

Session 7b: Students Who Didn't Save Their Code

Some students will show up to race day without code. Options:

  1. Check Google Classroom — did they save a draft in Assignment 7?
  2. If not, give them the starter code from the slides and 10 minutes during tuning time
  3. They probably won't win, but they can still compete — a slow clean run beats no run

Vincent and Winston in Sessions 7a–7b

Session 7a: They should use a for loop and finish the square quickly. Then redirect them to tuning for speed. Challenge: "Can you get your time under 10 seconds and still stay in the lane?" They should also be experimenting with velocity and turn timing to find the limit.

Session 7b: Have them run first. Their fast, clean run shows the rest of the class what's possible and sets the benchmark. After their runs, they can help struggling students diagnose turn issues — but remind them: point and explain, don't type for the other student.

If a Student Was Absent for Session 7a

They missed the build session. In Session 7b:

  1. Give them the starter code directly (show them slide 4 from Session 7a)
  2. They get the full 10-minute tuning window to build their square
  3. Focus them on getting a basic square working — skip the blinker if they're running out of time (a slow clean run with all signals is better than a fast run with no signals)
  4. They can still compete, just from behind. The unlimited attempts policy means they can iterate during the race.

7. Quick Reference: Key Code for Square Up

Keep these handy during both sessions for quick student help:

# Drive forward (one side of the square)
await motor_pair.move_for_time(motor_pair.PAIR_1, 2000, 0, velocity=50)
#                                                 ^^^^       ^^^^
#                                           time (ms)    speed

# Turn right 90 degrees
await motor_pair.move_for_time(motor_pair.PAIR_1, 800, 100, velocity=30)
#                                                 ^^^  ^^^
#                                            time    steering (100 = hard right)

# Right blinker function
async def right_blinker():
    for y in range(5):
        light_matrix.set_pixel(4, y, 100)    # column 4 = rightmost
    await runloop.sleep_ms(500)
    light_matrix.clear()

# Beep
await sound.beep(800, 300, 100)    # frequency, duration, volume

# For loop version (Green Table)
for i in range(4):
    await motor_pair.move_for_time(motor_pair.PAIR_1, 2000, 0, velocity=50)
    await right_blinker()
    await sound.beep(800, 300, 100)
    await motor_pair.move_for_time(motor_pair.PAIR_1, 800, 100, velocity=30)

Tuning Cheat Sheet

What to ChangeEffectTypical Range
Drive time (ms) How far the robot drives per side 1500–3500 ms (depends on velocity)
Drive velocity How fast the robot drives 30–80 (higher = faster but less control)
Turn time (ms) How far the robot rotates 600–1200 ms (this is the most sensitive parameter)
Turn velocity How fast the robot turns 20–40 (keep this low for precision)
Steering on straights Corrects drift (0 = straight, positive = right, negative = left) -5 to 5 (small corrections only)
The #1 tuning tip: Tune the turn first. A bad turn compounds over 4 corners. If the turn is even 5° off, the robot will be 20° off by the end of the square. Get the turn to exactly 90° before worrying about drive distance.
